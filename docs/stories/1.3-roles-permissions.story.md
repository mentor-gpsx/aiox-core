# Story 1.3: Implement Roles & Permissions Matrix

## Story Statement

Create role-based permission matrix (5 roles × 7 resources × 4 actions). Implement `@CheckPermission()` decorator for NestJS controllers to enforce access control at endpoint level.

## Roles & Permissions

| Role | Sales | Commissions | Users | Accounts |
|------|-------|-------------|-------|----------|
| ADMIN | CRUD | CRUD | CRUD | CRUD |
| COMERCIAL | CR | R | — | — |
| FINANCEIRO | R | RU | — | RU |
| GESTOR | R | R | — | — |
| AUDITOR | R | R | — | — |

**Legend:** C=Create, R=Read, U=Update, D=Delete

## Acceptance Criteria

- [x] `permissions` table seeded with full matrix (migration 002_permissions.sql)
- [x] `@CheckPermission('sales', 'CREATE')` decorator works (SetMetadata implementation)
- [x] Guard validates permission before controller executes (PermissionGuard + @UseGuards)
- [x] Unauthorized access returns HTTP 403 (ForbiddenException in guard)
- [x] Dynamic permission grant/revoke (ADMIN only) (POST /permissions/grant, DELETE /permissions/revoke)
- [x] Tests verify role boundaries (22 unit tests covering all 5 roles × resources)

## Implementation

1. **Database:** Seed permissions matrix via migration
2. **Guard:** `PermissionGuard` + `@CheckPermission()` decorator
3. **Service:** Permission checking logic
4. **Tests:** Unit tests for each role/resource combination
